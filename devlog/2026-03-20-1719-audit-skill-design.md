# /audit Skill 设计方案：业务逻辑审计

## 背景

在 ops-analytics 项目中，实施机器人跌落统计功能后，通过事后的跨组件审查发现了 3 个 `/new-feature` 自检阶段未能捕获的 bug：

1. `_normalize_progress` 丢弃 cleaning_gear / gear_change_reason（数据链路断裂）
2. 中断记录的 robot_id / reporter_id / response_time_min 未保存到 DB（保存流程缺失）
3. 跌落统计时长直接求和未合并重叠区间，与 KPI 面板数值矛盾（计算不一致）

其中第 3 个是关键级别 — 如果上线会直接误导绩效判断。

## 为什么需要独立 skill

现有 skill 的覆盖范围：

| Skill | 视角 | 盲区 |
|-------|------|------|
| `/new-feature` 阶段三（方案自评） | 被修改函数的调用方是否受影响 | 不追踪字段在 normalize→save 之间是否断裂 |
| `/new-feature` 阶段六（实施自检） | 改动代码本身的正确性 | 不检查同一指标在不同展示位置的计算一致性 |
| `/regression-check` | 功能能否跑、是否 break 已有功能 | 不做数值精确性审计 |

`/audit` 填补的是「跨组件数据正确性」这个盲区：
- 字段是否在中间环节断裂
- 同一数值在不同位置的算法是否一致
- 异常数据是否被可追踪地处理

## 定位与触发场景

**定位**：部署前的数据链路和计算正确性审计

**适用场景**：
- 新增/修改了 AI prompt 中的输出字段
- 改动了数据解析（result_parser）、存储（database）、展示（ui）中的任意环节
- 涉及数值计算（时长、面积、效率、占比）的变更

**不适用**：纯 UI 样式调整、配置变更等不涉及数据流的改动

## Skill 内容

文件路径：`C:\Users\Administrator\.claude\commands\audit.md`

```markdown
对最近的改动执行业务逻辑审计，追踪数据从源头到展示的完整链路，确保计算精确一致。

审计范围：$ARGUMENTS

如果未指定范围，则基于 git log 识别最近的改动自动确定审计范围。

---

## 步骤一：识别审计目标

1. 通过 `git log` 和 `git diff` 确定最近改动涉及的数据字段和计算逻辑
2. 列出需要审计的数据流（例如：字段 X 从 AI 输出 → parser → DB → UI）

## 步骤二：数据链路完整性检查

逐环节追踪每个相关字段：

1. **AI prompt**（`report_analyzer.py`）：是否要求输出该字段？
2. **规范化**（`result_parser.py`）：`_normalize_*` 函数是否保留该字段？
3. **存储**（`database.py`）：`add_*` 函数签名和 INSERT 语句是否包含该字段？`save_structured_to_db` 是否传递该字段？
4. **回填**（`database.py`）：需要 ID 映射的字段（如 name → id）是否有回填逻辑？
5. **查询**（`database.py`）：`get_*` 查询是否 JOIN 了相关表以返回该字段？
6. **展示**（`ui/*.py`）：前端是否正确读取并展示该字段？

**任一环节断裂 → 记录为 bug，标注严重度**

## 步骤三：计算一致性检查

对涉及数值计算的指标：

1. 同一指标在不同位置（KPI 面板、明细表、汇总表）的计算方式是否一致？
2. 涉及时间段聚合的计算，是否正确处理了重叠区间？
3. 百分比、占比的分母是否与展示语义匹配？
4. 浮点精度：round 的时机和精度是否合理？

## 步骤四：边界情况检查

1. 空值 / 缺失数据：字段为空时是否有合理兜底？
2. 零值：除数为零是否会导致异常？
3. 异常数据（如 end < start）：是否被捕获？是否有日志记录？
4. 数据被静默丢弃时，是否可追踪？

## 步骤五：输出审计报告

1. 将发现整理为结构化报告，按严重度分类：
   - **关键**：计算错误、数据丢失、指标矛盾 → 必须修复
   - **中等**：静默丢弃、精度损失、可追踪性不足 → 建议修复
   - **低**：展示不一致但不影响决策 → 可后续优化
2. 对每个问题给出：位置、问题描述、影响场景、修复建议
3. 列出已验证无误的环节
4. 询问用户是否立即修复关键和中等问题

**发现关键问题 → 阻塞部署，必须修复后才能上线**

---

## 原则

- 审计关注的是「数据正确性」，不是「代码风格」或「功能可用性」
- 对涉及绩效、考核、计费的数值，零容忍计算误差
- 每个结论必须有代码位置和具体场景支撑，不做笼统判断
```

## 与其他 skill 的协作

典型工作流：
1. `/new-feature` — 实施功能
2. `/audit` — 审计数据正确性
3. 修复审计发现的问题
4. `/deploy` — 部署上线
