# Devlog 开发日志规范

对任何项目，当需要进行非平凡的问题分析、方案设计、架构评估时，将分析过程和结论写入项目根目录的 `devlog/` 目录。

## 何时创建 devlog

- 需要定义问题、分析根因、设计解决方案的任务
- 涉及多文件修改的实现计划
- 架构评估、技术选型、方案对比
- 重要的 bug 修复方案（需记录调查过程和修复策略）
- 不适用：单行修复、简单重命名等无需分析的琐碎改动

## 文件命名

格式：`YYYY-MM-DD-HHMM-语义描述.md`

- 日期和时间取文档创建时的实际时间（含小时分钟）
- 语义描述用小写英文短横线连接，简明概括主题
- 示例：`2026-03-03-1530-storage-analysis.md`、`2026-03-02-1912-cleanup-legacy-password.md`

## 文档内容

不强制固定模板——根据问题性质灵活组织，但通常包含：

1. **问题/背景**：要解决什么问题，为什么需要处理
2. **调查/分析**：现状是什么，根因是什么，影响范围多大
3. **方案设计**：怎么解决，涉及哪些文件，关键代码变更
4. 可选：验证计划、已知限制、未来扩展方向

## 工作流程

1. 开始分析时先在 `devlog/` 创建文档，边分析边记录
2. 方案确定后作为实施参考，实施过程中如有偏差及时更新
3. 完成后文档留存，作为项目决策的历史记录
