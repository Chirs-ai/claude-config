针对用户提出的需求，执行完整的「分析 → 设计 → 实施 → 验证 → 提交」开发流程。

需求内容：$ARGUMENTS

---

## 阶段一：可行性分析

1. 根据需求关键词搜索并阅读相关源文件，理清数据结构、调用链、渲染逻辑
2. 判断现有数据和架构是否支持该需求
3. 给出明确结论：可行 / 有条件可行 / 不可行
4. **不可行 → 终止流程，向用户说明原因**
5. **有条件可行 → 列出前置条件或不确定点，请用户澄清后再继续**

## 阶段二：方案设计

1. 列出需要修改的文件和函数，描述每处具体变更
2. 说明边界情况处理方式
3. 评估改动量级

## 阶段三：方案自评

1. 搜索所有调用了被修改函数/模块的地方
2. 逐一评估每个调用场景是否受影响
3. 检查函数签名、返回值、数据流是否有破坏性变更
4. **发现设计缺陷 → 回到阶段二修正方案**
5. **有不确定的地方 → 列出问题，请用户澄清后再继续**

## 阶段四：保存方案到 devlog

1. 通过 `date '+%Y-%m-%d-%H%M'` 获取实际当前时间
2. 在项目 `devlog/` 目录创建文档，文件名格式：`YYYY-MM-DD-HHMM-语义描述.md`
3. 将分析结论、方案设计、影响评估整理为结构化文档写入
4. 向用户展示方案摘要，等待用户批准

**用户未批准 → 根据反馈调整方案，更新 devlog，重新请求批准**

## 阶段五：实施

1. 按方案逐步修改代码
2. 实施过程中如发现方案需要调整，先更新 devlog 再继续

## 阶段六：实施自检

1. 重新审查所有改动的代码，检查：
   - 逻辑正确性（边界情况、空值处理）
   - 是否会 break 已有功能（回归风险）
   - 是否引入潜在风险（安全、性能、兼容性）
2. **发现问题 → 立即修复，修复后重新自检**

## 阶段七：测试

1. 如果项目有测试框架，运行相关测试
2. 如果没有自动化测试，尝试启动应用验证功能是否正常
3. **测试失败 → 分析原因，修复后重新测试**
4. **所有测试通过 → 进入提交阶段**

## 阶段八：提交

1. 使用 /gitpush 提交代码

---

## 流程控制原则

- 每个阶段的检查点如果不通过，不得跳到下一阶段
- 需要用户介入时，清晰列出问题并等待回复，不要自行假设答案
- 整个流程中保持输出简洁，只在关键节点汇报进展
